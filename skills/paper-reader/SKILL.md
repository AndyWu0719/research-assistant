# Paper Reader

## 何时使用

- 用户给出一篇具体论文，希望做深入解读。
- 用户需要知道“这篇论文到底做了什么、为什么有效、值不值得复现”。
- 用户希望把复杂方法转成通俗解释或图示。

## 输入

- 论文标题、链接、PDF 或唯一可识别的信息
- 用户关注点，例如：方法细节、实验设计、创新性、复现价值
- 用户背景与约束，例如：是否是入门者、是否打算单卡复现、时间是否紧

## 工作流

1. 确认论文对象和版本，避免混淆同名工作。
2. 先提炼一句话结论，再拆解：
   - 研究问题
   - 核心假设
   - 方法组成
   - 主要实验结论
3. 把方法还原成可理解的流程，优先用 ASCII 或 Mermaid 图说明。
4. 区分作者声称的贡献与论文真正被实验支持的部分。
5. 提炼优势、局限、失败模式、适用边界。
6. 如果用户给了现实约束，增加“在当前约束下能否复现/迁移”的判断。
7. 尽量补齐代码、模型、数据、项目页链接。

## 输出要求

输出至少包含以下部分：

1. `一句话总结`
2. `论文速览`
3. `核心问题与方法`
4. `通俗图示`
5. `关键实验与结论`
6. `优点 / 局限 / 适用边界`
7. `复现门槛`
8. `开源资源`

如果任务来自本地网页，且 prompt 提供了明确输出路径，则结果应写入：

- Markdown：`outputs/paper_summaries/<name>-zh.md`
- JSON sidecar：同名 `.json`

推荐写法：

- `一句话总结`：这篇论文解决什么问题，用什么办法，最重要的结果是什么。
- `通俗图示`：把输入、模块、训练目标、输出关系画出来。
- `复现门槛`：说明数据、算力、工程复杂度、最可能踩坑的地方。

## 深读要求

- 不要只复述摘要，要解释“机制为什么可能有效”。
- 要指出论文最依赖哪些前提条件。
- 对实验结果，优先看：
  - 和哪些基线比
  - 提升是否稳定
  - 是否有关键消融
  - 是否有可疑设定或隐藏代价
- 如果论文有明显缺口，要明确写出，而不是只做礼貌性总结。

## 通俗化要求

- 默认用中文解释。
- 对陌生术语，优先给一句通俗重述。
- 能图示就图示，优先说明信息流、训练流或决策流。

## 开源与复现

- 能给开源地址就给开源地址。
- 如果没有公开代码、模型或数据，要明确标注。
- 若用户有资源限制，要补充“低成本复现版本”建议，例如：
  - 缩小数据规模
  - 先跑公开 checkpoint
  - 先复现核心模块而不是完整系统

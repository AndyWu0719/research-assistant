# Idea Feasibility

## 何时使用

- 用户提出一个研究想法，想知道值不值得做。
- 用户需要判断：是否已有类似工作、创新点是否站得住、最容易失败在哪里。
- 用户希望得到一个最小可行验证路径，而不是泛泛而谈的鼓励或否定。

## 输入

- 研究想法的描述
- 目标问题、目标数据或目标应用场景
- 现实约束：算力、时间、数据、团队、工程能力
- 额外目标：发论文、做项目、做 demo、做产品验证

## 工作流

1. 先把想法改写成更可评估的形式：
   - 问题是什么
   - 核心假设是什么
   - 方法变化点是什么
   - 期望带来什么收益
2. 如果用户提供了约束，先做约束感知分析，判断这个想法在现实中是否跑得动。
3. 找最近邻相关工作和最强基线，区分：
   - 同问题同方法
   - 同问题不同方法
   - 不同问题但机制相似
4. 判断创新空间：
   - 是新问题、新信号、新目标、新组合，还是已有套路的小变体
5. 判断主要风险：
   - 创新不足
   - 实验难验证
   - 数据不可得
   - 计算成本过高
   - 工程复杂度超标
6. 设计最小可行验证路径：
   - 最低成本实验
   - 必要 baseline
   - 最小数据与最小指标
   - 最需要做的消融
7. 给出继续做、收缩做、转向做或放弃做的明确建议。

## 输出要求

输出至少包含以下部分：

1. `想法重述`
2. `最相关工作`
3. `创新空间判断`
4. `主要风险`
5. `可行性结论`
6. `最小可行验证路径`
7. `推荐论文 / 代码 / 数据`

如果任务来自本地网页，且 prompt 提供了明确输出路径，则结果应写入：

- Markdown：`outputs/feasibility_reports/`
- JSON sidecar：同名 `.json`

## 结论类型

优先给出明确判断，而不是模糊表态：

- `值得做`
- `值得做，但要收缩问题`
- `可以做成工程/应用验证，但学术创新偏弱`
- `当前约束下不建议优先做`

## 风险分析要求

- 每个风险后面尽量给一个对应缓解动作。
- 如果风险来自约束不匹配，要明确说明是哪个约束造成的。
- 如果想法本质上只是已有方法的轻微拼接，要直接指出，不要包装成“可能有潜力”。

## 最小可行验证路径要求

- 尽量给低成本方案，优先 1 个核心实验、1 个强 baseline、1 组关键消融。
- 优先设计能尽早证明或证伪核心假设的实验。
- 能给开源实现、可替代数据、可直接改的 baseline，就给出来。

## 通用要求

- 默认使用中文输出。
- 输出尽量结构化、清晰、易理解。
- 能图示就图示，例如用二维矩阵展示“创新性 x 可行性”。
- 排序或优先级判断必须解释依据。
- 用户给出约束时，优先做约束感知分析。

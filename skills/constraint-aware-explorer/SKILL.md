# Constraint Aware Explorer

## 何时使用

- 用户明确给出算力、显存、时间、预算、数据或团队约束。
- 用户的问题核心是“在这些限制下我到底该做什么”。
- 用户需要的是低成本、可启动、可验证的研究切入点。

## 输入

- 硬约束：显存、GPU 数量、预算、时间、数据可得性、部署限制
- 软偏好：方向兴趣、期望创新度、是否偏论文、是否偏工程
- 已有资源：现成代码、现成数据、熟悉的框架、已有 baseline

## 核心原则

- 先判断能不能做，再判断值不值得做。
- 优先推荐低启动成本、低失败成本、反馈周期短的方向。
- 不把“理论上更强但现实上做不动”的方向排在前列。
- 默认使用 `resource-constrained` 排序画像；只有在用户明确要求追热点时，才混入 `trend-focused`。

## 工作流

1. 拆分硬约束与软约束。
2. 画出可行边界：哪些方向基本不可做，哪些方向可低成本启动。
3. 生成候选方向池，并按资源匹配度优先过滤。
4. 对每个方向补齐：
   - 代表论文
   - 开源代码
   - 可用数据
   - 最低成本起步方案
5. 按 `resource-constrained` 排序，明确说明为什么适合当前用户。
6. 给出低成本路线图，以及至少一组“不建议优先做”的方向。

## 输出要求

输出至少包含以下部分：

1. `约束摘要`
2. `可行性边界`
3. `推荐方向榜`
4. `每个方向的切入论文与开源资源`
5. `最低成本验证路径`
6. `不建议优先做的方向`

如果任务来自本地网页，且 prompt 提供了明确输出路径，则结果应写入：

- Markdown：`outputs/constraint_reports/`
- JSON sidecar：同名 `.json`

建议表格列：

| 列名 | 含义 |
| --- | --- |
| 排名 | 推荐顺序 |
| 方向 | 研究切入点 |
| 为什么适合当前约束 | 资源匹配原因 |
| 入门论文 | 最适合先看的论文 |
| 开源资源 | 代码、数据、项目页 |
| 最低成本实验 | 可以立刻开始的验证动作 |
| 风险 | 可能卡住的点 |

## 路线图要求

- 尽量输出 2-4 周可执行的低成本路径。
- 先做信号验证，再做复杂扩展。
- 如果数据有限，优先推荐：
  - 可直接使用公开数据的方向
  - 可从预训练模型迁移的方向
  - 可基于已有开源 baseline 微调或裁剪的方向

## 质量要求

- 不能只给“大方向”，要给具体可落地入口。
- 能给开源地址就给开源地址。
- 排序必须解释依据，尤其要解释为什么排除了某些更热门方案。
- 如果用户条件非常紧，要明确指出“能做的上限”和“当前不现实的部分”。

## 通用要求

- 默认使用中文输出。
- 输出尽量结构化、清晰、易理解。
- 能图示就图示，例如用路径图或 trade-off 图表示选择空间。
- 优先给可启动的论文、代码和低成本实验。

# Topic Mapper

## 何时使用

- 用户给出一个方向，希望知道“这个方向周边有哪些论文”。
- 用户不只要 Top papers，还要知道论文之间的层级关系和邻近关系。
- 用户希望获得一张可导航的主题地图，而不是单一榜单。

## 输入

- 研究方向、任务、问题定义或关键词
- 是否更关注核心工作、邻近方法、基础工作或应用落地
- 排序偏好与现实约束

## 分层定义

- `Tier 1`：直接相关。解决同类问题、同类设定、同类目标。
- `Tier 2`：强相关邻近。方法可迁移、设定相近、对当前方向有直接借鉴价值。
- `Tier 3`：背景/基础。奠基工作、通用技术、评价框架或关键数据集论文。

## 工作流

1. 把用户方向收敛成一个明确的核心问题。
2. 提取关键子主题、常见别名和邻近问题。
3. 按主题聚类，再按相关性放入不同 Tier。
4. 每个 Tier 内部再根据 `configs/ranking_profiles.md` 排序。
5. 输出“主题图 + 分层榜单 + 阅读路径”三件套。
6. 如果用户给了现实约束，额外标记哪些分支更容易低成本进入。

## 输出要求

输出至少包含以下部分：

1. `方向定义`
2. `主题分层图`
3. `Tier 1 / Tier 2 / Tier 3 论文清单`
4. `层内排序依据`
5. `阅读路径建议`
6. `开源资源`

如果任务来自本地网页，且 prompt 提供了明确输出路径，则结果应写入：

- Markdown：`outputs/topic_maps/`
- JSON sidecar：同名 `.json`

建议图示形式：

```text
核心方向
  ├─ Tier 1: 直接相关
  ├─ Tier 2: 可迁移邻近
  └─ Tier 3: 背景与基础
```

## 排序要求

- 先解释“为什么属于这个层级”，再解释“为什么在层内排这个位置”。
- 如果某论文热度很高但只能算邻近工作，不能错误放进 Tier 1。
- 范围过大时，先做主题簇，再在每个簇内分层。

## 质量要求

- 每层都应说明该层对用户的价值。
- 不要把纯背景论文堆满前排，导致直接相关工作被淹没。
- 能给论文、代码、数据、项目页就给。
- 如果约束明显，优先指出“适合入手”的子分支，而不是只画完整地图。

## 通用要求

- 默认使用中文输出。
- 输出尽量结构化、清晰、易理解。
- 能图示就图示。
- 排序必须解释依据。
- 用户给出约束时，优先做约束感知分析。

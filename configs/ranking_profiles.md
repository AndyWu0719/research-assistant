# Ranking Profiles

本文件定义 research-assistant 默认使用的排序画像。凡是输出 Top K、分层清单、阅读顺序或推荐顺序，都应显式说明使用了哪个 profile，以及为什么适合当前任务。

## 通用规则

- 每个维度按 `1-5` 打分，`5` 为最好。
- 最终分数按加权和计算，权重总和为 `100`。
- 用户明确给出排序偏好时，优先服从用户偏好。
- 用户给出现实约束时，默认优先使用 `resource-constrained`。
- 如果需要混合画像，必须说明混合方式，例如：`resource-constrained 70% + trend-focused 30%`。

## 通用维度定义

| 维度 | 含义 |
| --- | --- |
| `relevance` | 与用户问题、任务设定、数据设定、目标指标的直接相关性 |
| `evidence-quality` | 方法与实验是否严谨，是否有充分对比、消融和误差分析 |
| `impact` | 学术或社区影响力，如代表性、引用、采用度、基准地位 |
| `recency` | 新近程度，是否覆盖近年方向变化 |
| `openness` | 是否有公开代码、模型、数据、文档，复现阻力是否低 |
| `trend-signal` | 趋势热度，是否处于活跃增长、讨论度高、后续工作多 |
| `resource-fit` | 与用户现实约束的匹配程度，包括算力、数据、时间和工程复杂度 |

## balanced-default

### 适用场景

- 用户没有特别强调“追新”或“低成本”。
- 需要一个兼顾相关性、质量、影响力和可复现性的默认榜单。
- 适合入门梳理、综述初筛、一般性的论文推荐。

### 权重

| 维度 | 权重 |
| --- | --- |
| `relevance` | 30 |
| `evidence-quality` | 20 |
| `impact` | 15 |
| `recency` | 10 |
| `openness` | 10 |
| `resource-fit` | 10 |
| `trend-signal` | 5 |

### 排序逻辑

- 先保证和用户任务强相关，再看论文是否扎实、是否具有代表性。
- 不盲目追最新，也不只看老经典。
- 对新论文会给时效性加分，但不会因为“新”而压过明显更相关、更可靠的工作。

### 同分处理

1. `relevance`
2. `evidence-quality`
3. `openness`
4. `recency`

### 输出时必须说明

- 为什么这个默认画像适合当前问题。
- Top 3 为什么排在前面。
- 如果榜单中既有经典论文又有新论文，要说明二者各自承担什么角色。

## trend-focused

### 适用场景

- 用户明确强调“最新进展”“最近两年热点”“领域风向”“值得追的方向”。
- 目标是把握研究趋势，而不是只找最稳的经典论文。
- 适合 topic scouting、热点追踪、做选题前的快速风向判断。

### 权重

| 维度 | 权重 |
| --- | --- |
| `relevance` | 22 |
| `trend-signal` | 25 |
| `recency` | 20 |
| `impact` | 13 |
| `evidence-quality` | 8 |
| `openness` | 7 |
| `resource-fit` | 5 |

### 排序逻辑

- 优先找近期活跃、后续工作多、社区关注度高的论文。
- 相比默认画像，更愿意把“新而有潜力”的工作排到前面。
- 仍需避免把和用户主题关系较弱、只是“很热”的论文抬得过高。

### 同分处理

1. `trend-signal`
2. `recency`
3. `relevance`
4. `openness`

### 输出时必须说明

- 趋势信号来自哪里，例如近期大量后续工作、活跃 benchmark、广泛复现。
- 热度是否已经转化为稳定价值，还是仍有泡沫风险。
- 如果榜单偏新，要补一句“补课入口”，指出哪几篇基础论文应该先读。

## resource-constrained

### 适用场景

- 用户明确说算力有限、单卡 24G、数据有限、预算紧、时间短、团队小。
- 任务目标不是“理论上最强”，而是“在现实约束下最值得做”。
- 适合选题裁剪、低成本复现、工程可落地路径设计。

### 权重

| 维度 | 权重 |
| --- | --- |
| `resource-fit` | 32 |
| `openness` | 18 |
| `relevance` | 18 |
| `evidence-quality` | 12 |
| `recency` | 8 |
| `impact` | 7 |
| `trend-signal` | 5 |

### 排序逻辑

- 先看能不能做，再看值不值得做。
- 强调资源匹配、可复现性、启动成本和失败代价。
- 即使某论文很强，如果对用户当前资源明显不友好，也不应排在前列。

### 资源匹配度拆解

评估 `resource-fit` 时，优先考虑：

1. 显存和训练成本是否匹配
2. 数据是否容易获得或替代
3. 工程复杂度是否适合当前团队
4. 实验周期是否符合时间窗口
5. 是否存在低成本 baseline 或公开实现可直接起步

### 同分处理

1. `resource-fit`
2. `openness`
3. `relevance`
4. `evidence-quality`

### 输出时必须说明

- 用户约束是什么，以及它们如何影响排序。
- 每个高排名项为什么“能做得动”。
- 至少指出 1-3 个不建议优先做的方向，并说明不适配的原因。

## 排序说明模板

实际输出时，至少包含以下信息：

```text
排序画像: balanced-default / trend-focused / resource-constrained / 自定义混合
用户约束: ...
关键维度: ...
Top 3 排名前置原因: ...
并列打破方式: ...
```
